# 코딩 규칙

이 문서는 이 저장소의 프로덕션 코드를 작성하거나 수정할 때 따르는 기준입니다.

## 아키텍처

- NestJS 구조와 의존성 주입 패턴을 사용합니다.
- controller는 얇게 유지합니다.
- 비즈니스 로직은 service로 이동합니다.
- 모든 데이터베이스 접근은 repository에 둡니다.
- 모듈 간 도메인 격리를 유지합니다.
- 직접적인 Express 요청 처리 패턴은 사용하지 않습니다.
- 편의성보다 명시적인 의존 경계를 우선합니다.
- 모듈의 외부 API는 안정적이고 테스트하기 쉬워야 합니다.

## 모듈 파일 구성 규칙

모듈 내부 파일은 다음 형태를 기본으로 합니다.

- `<domain>.controller.ts`
- `<domain>.service.ts`
- `<domain>.repository.ts`
- `<domain>.module.ts`
- `<domain>.dto.ts`

모듈이 커지더라도 같은 소유권 구조를 유지하며 분리합니다.

## 설정

- 설정은 `.env`에서 읽습니다.
- DB를 사용하는 런타임 경로에서는 `DATABASE_URL`을 필수로 둡니다.
- 비밀값, URL, 포트, 환경별 값은 하드코딩하지 않습니다.
- 환경 변수 접근은 중앙화하고, 필수 설정은 초기에 검증합니다.
- 흩어진 문자열 접근보다 타입 기반 config 접근을 선호합니다.

## Prisma schema

- schema 정의에는 비즈니스 로직을 넣지 않습니다.
- 모델명과 필드명은 명확하게 유지합니다.
- 자주 조회되는 경로의 필드는 인덱스를 고려합니다.
- schema 변경은 repository의 실제 쿼리 요구와 맞물리게 관리합니다.

기본 모델 스타일 예시는 다음과 같습니다.

```prisma
model User {
  id        Int      @id @default(autoincrement())
  email     String   @unique
  createdAt DateTime @default(now())
}
```

## 데이터베이스와 Prisma

- 데이터베이스는 PostgreSQL만 사용합니다.
- Prisma ORM은 repository 클래스를 통해서만 사용합니다.
- controller나 service에서 Prisma를 직접 주입하거나 호출하지 않습니다.
- 쿼리 의도는 읽기 쉽고, 해당 도메인 repository 범위 안에 있어야 합니다.
- Prisma 트랜잭션은 repository 레이어에서만 사용합니다.
- 여러 쿼리를 하나의 흐름으로 묶을 때는 callback 기반 transaction 형태를 사용합니다.
- repository에는 비즈니스 로직을 넣지 않습니다.
- 복잡한 쿼리는 service가 아니라 repository에 둡니다.

트랜잭션 예시는 다음과 같습니다.

```ts
await this.prisma.$transaction(async (tx) => {
  // ...
});
```

## 성능

- N+1 쿼리 패턴을 피합니다.
- 더 좁은 결과로 충분하면 `include`보다 `select`를 우선합니다.
- API 응답 요구에 맞게 쿼리 형태를 의도적으로 설계합니다.
- 응답 payload는 제한하고, 불필요한 필드는 반환하지 않습니다.
- 목록 엔드포인트에는 페이지네이션, 필터링, 명시적 정렬을 우선 적용합니다.
- 무제한 동시성, 반복적인 원격 호출, 큰 메모리 변환에 주의합니다.
- 최적화는 먼저 소유권과 쿼리 의도를 명확히 한 뒤 진행합니다.

## API 계약

- REST 엔드포인트만 노출합니다.
- 버전 경로는 `/api/v1/...`를 사용합니다.
- 모든 요청 입력은 DTO 클래스를 사용합니다.
- 모든 인바운드 요청에 validation을 적용합니다.
- 응답 형태는 명시적이고 안정적으로 유지합니다.
- 응답은 항상 타입 객체를 반환합니다.
- 목록 API는 `limit`, `cursor` 페이지네이션을 지원합니다.
- 상태 코드, 에러 형태, 페이지네이션 계약은 일관되게 유지합니다.

## 보안

- 인증은 JWT를 사용합니다.
- 인증 관련 관심사는 전용 guard, strategy, decorator, auth service에 둡니다.
- 로그나 에러에 비밀값이나 토큰을 노출하지 않습니다.
- 외부 입력은 경계에서 검증하고 정제합니다.
- 인가 검사는 보호하려는 비즈니스 동작 가까이에 둡니다.

## 에러와 로깅

- 애플리케이션 전역 에러 처리는 global exception filter를 사용합니다.
- 중앙화된 logger를 사용합니다.
- 상세 쿼리 로깅은 개발 환경에서만 허용합니다.
- 운영 환경 디버그 로그는 금지합니다.
- 에러는 구조화되어 디버깅 가능해야 합니다.
- DB나 인증 예외를 맥락 없이 삼키지 않습니다.
- 민감한 정보를 드러내지 않으면서도 추적 가능한 정도의 로그 문맥을 남깁니다.

## 테스트와 유지보수성

- 비즈니스 규칙은 HTTP transport 없이도 테스트 가능해야 합니다.
- service 로직은 unit test, 라우트/인증/영속성 계약은 integration 또는 e2e test를 우선합니다.
- 중요 엔드포인트는 validation, authorization, pagination, failure path를 함께 검증합니다.
- 동일한 동작이 여러 모듈에 반복되면 중복 제거를 검토합니다.
- 과도하게 영리한 추상화보다 읽기 쉬운 코드와 명시적인 이름을 우선합니다.

## JSDoc 작성 기준

- 모든 함수에 JSDoc을 강제하지는 않습니다.
- 아래 경우에는 함수별 또는 feature별로 필요한 만큼 JSDoc을 작성합니다.
- 공개 메서드의 입력/반환 계약이 중요할 때
- 인증, 인가, 트랜잭션, 캐시, 외부 연동처럼 부수 효과가 클 때
- 예외 조건이나 실패 케이스를 미리 알려야 할 때
- 코드만 읽고는 의도를 바로 이해하기 어려운 비즈니스 규칙이 있을 때
- feature 단위의 핵심 흐름을 설명해야 할 때

JSDoc은 짧고 실용적으로 유지합니다.

- 무엇을 하는지
- 어떤 입력을 기대하는지
- 어떤 값을 반환하는지
- 어떤 예외/주의사항이 있는지

## 마이그레이션과 시드

- schema 변경은 항상 Prisma migration 명령으로 관리합니다.
- 수동 DB 수정을 진실의 원천으로 삼지 않습니다.
- 필요한 초기 데이터가 있으면 seed 스크립트를 제공합니다.

## 런타임

- 서비스는 Docker 안에서 실행 가능해야 합니다.
- 컨테이너 밖 로컬 호스트 환경을 전제로 한 코드는 피합니다.
- 시작 시점에 핵심 설정과 의존성을 검증해 예측 가능한 부팅을 유지합니다.

## 금지 항목

- controller에 비즈니스 로직 작성
- controller에서 Prisma 직접 사용
- service에서 Prisma 직접 사용
- controller에서 repository 직접 호출
- controller에서 Prisma 직접 호출
- repository에 비즈니스 로직 작성
- service에 복잡한 쿼리 로직 작성
- 전역 mutable state 도입
