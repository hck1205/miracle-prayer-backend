# 작업 체크리스트

이 문서는 의미 있는 변경을 전달하기 전 확인해야 할 체크리스트입니다.

## 수정 전

- 어떤 NestJS 모듈이 이 변경을 소유하는지 먼저 식별합니다.
- 관련 파일을 먼저 읽고, 일반론보다 현재 저장소의 로컬 패턴을 우선 보존합니다.
- 변경이 `controller -> service -> repository -> Prisma` 흐름을 지키는지 확인합니다.
- Prisma 사용이 있다면 repository 레이어 안에 머무르는지 확인합니다.
- 트랜잭션이 있다면 repository 레이어 안에 있는지 확인합니다.
- schema나 migration 변경이 있다면 Prisma migration 흐름으로 처리하는지 확인합니다.
- 목록 API, 인증, 쓰기 경로처럼 성능 민감한 영역인지 확인합니다.
- worktree에 이미 존재하는 사용자 변경을 덮어쓰지 않는지 확인합니다.

## 구현 중

- 편집 범위는 요청된 결과에 맞게 최소화합니다.
- NestJS 네이티브 구조와 네이밍을 우선합니다.
- controller는 요청/응답 처리에만 집중시킵니다.
- service는 비즈니스 로직에 집중시킵니다.
- repository는 Prisma를 통한 PostgreSQL 접근에만 집중시킵니다.
- repository에는 비즈니스 로직을 넣지 않습니다.
- 의도가 바로 드러나지 않을 때만 짧은 주석을 추가합니다.
- `/api/v1/...` 버전 규칙을 유지합니다.
- 요청 입력에는 DTO와 validation을 적용합니다.
- 목록 엔드포인트에는 필요 시 `limit`, `cursor` 페이지네이션을 넣습니다.
- 응답은 계속 타입 객체 형태를 유지합니다.
- N+1 패턴을 피하고 가능하면 `select`를 우선합니다.
- 상태 코드, 에러 형태, 응답 계약을 일관되게 유지합니다.
- 설정은 `.env` 기반으로 유지하고 Docker 호환성을 해치지 않습니다.
- feature별 핵심 흐름이나 함수별 계약이 복잡할 때는 필요한 만큼 JSDoc을 추가합니다.
- 특히 public 메서드, 인증/인가 처리, 트랜잭션 경계, 예외 규칙이 중요한 함수는 JSDoc 작성을 우선 검토합니다.

## 전달 전

- 가능한 범위에서 가장 작고 의미 있는 검증을 실행합니다.
- controller, service, repository 책임이 여전히 전역 규칙과 맞는지 확인합니다.
- auth, logger, exception handling 동작이 전역 규칙과 맞는지 확인합니다.
- schema, migration, seed 변경이 일관적인지 확인합니다.
- 성능 민감한 쿼리나 payload 결정이 있으면 명시합니다.
- 어떤 파일을 바꿨는지보다 어떤 동작이 바뀌었는지를 요약합니다.
- 누락된 테스트, 페이지네이션 엣지 케이스, 트랜잭션 리스크, migration 리스크, Docker/runtime 설정, 미해결 env 요구사항이 있으면 함께 알립니다.
